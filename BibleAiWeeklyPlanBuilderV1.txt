# Lyft Zone - Bible AI Weekly Plan Builder V1

## 1. Vision générale

Le AI Weekly Plan Builder V1 est le cerveau initial de création de weekly plans dans Lyft Zone.

Son rôle est de générer un weekly plan statique, éditable et publiable, basé sur le Training Profile, le User Exercise Pool, les priorités musculaires, les contraintes, la durée disponible et la doctrine de coaching Lyft Zone.

Il ne doit pas être confondu avec le futur AI Coach.

## 2. Distinction fondamentale

### AI Weekly Plan Builder V1

Crée un weekly plan draft.

Il:

* crée un plan hebdomadaire statique
* n’assigne pas le plan à une timeline
* ne crée pas de cycle multi-week automatiquement
* ne fait pas d’adaptation après logs
* ne fait pas de deload intelligent
* ne modifie pas automatiquement des plans existants
* ne remplace pas le futur AI Coach
* réutilise le Manual Builder pour l’édition finale

### AI Coach futur

Crée ou ajuste une vision long terme.

Il pourra:

* créer un cycle multi-week
* utiliser les workout logs
* comparer prévu vs réalisé
* proposer des ajustements
* proposer du deload
* proposer des substitutions
* faire évoluer le Training Profile avec validation utilisateur
* conserver une mémoire longitudinale plus avancée

## 3. Règle produit centrale

User Exercise Pool is a generation-time constraint, not a retroactive global validity rule.

Cela veut dire:

* Le User Exercise Pool contraint les nouvelles sélections générées par l’AI Builder.
* L’IA reçoit uniquement les exercices du pool.
* Après génération, le backend vérifie que tous les exerciseIds générés sont dans le même pool snapshot.
* Si un exerciseId généré est hors pool snapshot, le plan est rejeté avant la review et le Repair Loop. Le Repair Loop V1 est déclenché uniquement après une review initiale REPAIR_REQUIRED d’un plan ayant déjà passé les validations Schema, sémantiques, pool et business rules.
* Les plans existants ne sont pas revalidés contre le profil courant.
* Les drafts existants ne sont pas invalidés parce que le profil a changé.
* Les plans publiés ne sont pas invalidés parce que le profil a changé.
* Les anciens plans AI ne sont pas invalidés parce que le profil a changé.
* Les plans manuels gardent leur tolérance historique normale.

Le pool est donc une contrainte de décision au moment de la génération, pas une règle rétroactive.

## 4. Ordre de décision du AI Builder

L’ordre de décision officiel est:

1. Contraintes fixes

   * sessions per week
   * duration per session
   * User Exercise Pool
   * blocked constraints

2. Objectif principal

   * HYPERTROPHY uniquement dans le AI Weekly Plan Builder V1
   * STRENGTH et MIXED restent conservés dans le Training Profile, mais sont verrouillés pour la génération V1
   * Le backend refuse une génération AI Builder V1 pour STRENGTH ou MIXED

3. Priorités musculaires

   * primary focus
   * secondary focus
   * deprioritized area

4. Volume cible par muscle

5. Fréquence par muscle

6. Split choisi

7. Séances

8. Exercices

9. Sets, reps, RIR, rest, tempo

10. Validation backend

Cet ordre doit guider le prompt IA, la doctrine coach, le ProgramGenerationContext, la review IA et les validations.

## 5. Non négociables

Le AI Weekly Plan Builder V1 doit respecter ces règles:

* Le backend reste la source de vérité.
* Le frontend ne valide pas les règles métier critiques.
* L’IA ne peut pas inventer d’exercice.
* L’IA ne peut pas utiliser un exercice hors User Exercise Pool snapshot.
* L’IA ne peut pas contourner les contraintes blocked.
* Les caution signals restent soft.
* Le plan généré doit respecter la même structure qu’un weekly plan manuel.
* Le plan généré doit être éditable dans le Manual Builder.
* Le plan généré doit être publiable avec les règles normales du weekly plan.
* Le plan généré ne doit pas créer de cycle.
* Le plan généré ne doit pas apparaître dans la timeline tant qu’un cycle n’est pas créé plus tard.
* Le futur AI Coach doit pouvoir réutiliser les fondations sans les réécrire.

## 11. Coaching Notes et mémoire

### coachingNotesUser

Dans V1, le champ existant physicalNotes peut être utilisé comme base.

Décision recommandée:

V1:

* renommer l’UI en Coaching Notes
* garder le champ technique physicalNotes si on veut éviter une migration immédiate

Plus tard:

* migrer vers coachingNotesUser si on veut rendre le nom plus propre

Règles:

* visible par l’utilisateur
* éditable
* max 1000 à 1500 caractères
* ne doit pas répéter les champs structurés
* sert à ajouter du contexte non structuré

Exemples bons:

* préfère les séances simples
* aime garder certains exercices stables
* joue au tennis une fois par semaine
* préfère peu de changements de setup
* aime les machines pour les jambes

Exemples mauvais:

* primary goal hypertrophy
* intermediate
* 4 sessions per week
* focus upper chest
* blocked overhead pressing

Ces informations ont déjà des champs structurés.

### coachMemorySummary

Hors scope strict V1, mais à prévoir.

Règles:

* caché ou dev-only au départ
* généré par IA
* max 1000 à 1500 caractères
* ne doit pas répéter le Training Profile
* ne doit jamais contredire les champs structurés
* peut être compressé automatiquement quand il devient trop long

Règle de priorité:

Training Profile gagne toujours.
User Exercise Pool gagne toujours.
Backend validations gagnent toujours.
Coaching Notes et Coach Memory perdent en cas de conflit.

## 12. Output AI attendu

L’IA ne doit pas retourner du texte libre comme plan final.

Elle doit retourner un JSON strict.

Le texte libre est permis seulement pour:

* strategySummary
* planExplanation courte
* exercise notes rares
* reviewSummary
* warnings courts si nécessaires

### Sortie génération

L’output généré doit contenir:

* planName
* sessionsPerWeek
* strategySummary
* splitType
* workouts
* volumeTargets
* frequencyTargets
* progressionModel
* cautionHandling
* notesPolicy

### Workouts

Chaque workout doit contenir:

* name
* orderIndex
* estimatedDurationMinutes
* focus
* blocks

### Blocks

Types supportés V1:

* SINGLE
* SUPERSET si supporté proprement
* CARDIO si cardioRole le justifie

Garder GIANT_SET et CIRCUIT hors scope V1 même si le backend a des enums plus larges, sauf décision contraire.

### Exercises

Chaque exercice doit contenir:

* exerciseId
* exerciseName
* bodyParts
* muscleFocus
* defaultTempo
* defaultRestSeconds
* defaultTargetRir
* setTemplates
* notes seulement si pertinent

Chaque set doit contenir:

* setIndex
* setType
* targetRir
* tempo
* restSeconds
* une seule des deux prescriptions de répétitions suivantes:

Prescription exacte:

* targetReps: entier
* minReps: null
* maxReps: null

Prescription par plage:

* targetReps: null
* minReps: entier
* maxReps: entier
* minReps doit être inférieur ou égal à maxReps

Il est interdit de combiner targetReps avec minReps/maxReps.
Tous les champs restent présents dans le JSON strict, même lorsqu’ils valent null.

### Cardio

Si cardio est généré:

* blockType CARDIO
* exactement un exercice
* aucun setTemplate
* cardioPrescription valide
* durationMinutes
* heartRateTargetMode
* machineSettings conformes à la modalité

## 13. Validation backend

La validation se fait en couches.

### Couche 1: JSON Schema AI

Valide la forme stricte de l’output fournisseur.

Le Schema impose notamment:

* champs requis
* champs inconnus interdits
* types et bornes
* block types V1
* contrat exclusif targetReps ou minReps/maxReps
* au moins un workout
* au moins un block par workout
* limites de cardinalité

### Couche 2: Validation sémantique AI

Valide les invariants relationnels qui ne sont pas entièrement exprimés par le JSON Schema:

* sessionsPerWeek égal à workouts.length
* orderIndex et setIndex séquentiels et uniques
* cardinalité exacte des blocs SINGLE, SUPERSET et CARDIO
* au moins un setTemplate pour les exercices de musculation
* minReps inférieur ou égal à maxReps
* égalité du nombre de sets dans un superset
* champs obligatoires de musculation non null
* structure cardio valide
* politique globale des notes

### Couche 3: AI normalization

Convertit l’output AI validé vers le document weekly plan normalisé.

Elle nettoie et organise notamment:

* strings
* nombres
* enums
* ordre des workouts, blocks, exercices et sets
* prescriptions cardio
* defaults d’exercice

### Couche 4: AI scoped pool validation

Valide que tous les exerciseIds générés sont dans allowedExerciseIds du même pool snapshot.

Cette validation est limitée au pipeline AI Builder.

### Couche 5: Weekly plan business rules

Valide:

* name
* sessionsPerWeek
* workout count
* workout names
* blocks
* block types
* setTemplates
* tempo
* rest
* RIR
* cardio blocks
* exerciseIds connus globalement
* publish readiness

### Couche 6: Analytics

Version actuelle: Weekly Plan Analytics V2

Calcule notamment:

* durée demandée par workout
* durée calculée par weeklyPlanMetrics
* différence calculée moins demandée
* ratio d’utilisation
* statut d’alignement de durée
* correctionRequiredWorkoutCount
* volume direct
* volume indirect séparé
* fréquence directe
* comparaison des volumeTargets
* comparaison des frequencyTargets
* couverture des metadata musculaires

### Couche 7: AI Program Review

Vérifie la cohérence de coaching.

### Couche 8: Repair

Répare une fois si nécessaire.

### Couche 9: Revalidation finale

Le backend réexécute les validations pertinentes avant toute sauvegarde.

Aucun plan invalide ne doit être persisté.

## 14. AI Program Review

La review IA est incluse dès V1.

Elle ne remplace pas la validation backend.

Elle vérifie:

* respect du primary goal
* respect des priorités musculaires
* cohérence du split
* volume cible vs volume généré
* fréquence par muscle
* durée estimée
* redondance d’exercices
* ordre des exercices
* équilibre global
* cohérence des supersets
* logique des RIR/rest/tempo
* gestion des caution signals
* absence de notes inutiles partout

Vocabulaire fatigue:

Ne pas exagérer overtraining.

Utiliser plutôt:

* fatigue locale excessive
* redondance inutile
* charge systémique élevée
* volume possiblement agressif selon le profil
* récupération à surveiller si le contexte le justifie

Ne pas suggérer qu’un utilisateur à 3 séances/semaine est automatiquement à risque.

## 15. Repair loop

V1 doit inclure un repair loop simple.

Pipeline:

1. Generate strict AI output
2. Validate JSON Schema
3. Validate AI semantic invariants
4. Normalize to weekly plan document
5. Validate against the generation-time pool snapshot
6. Validate weekly plan business rules
7. Calculate backend analytics
8. Run AI Program Review
9. If repairable issues exist, repair once:

La réparation V1 est déclenchée uniquement par une review initiale
REPAIR_REQUIRED.

Toutes les issues HIGH + REPAIRABLE sont obligatoires.

Les issues MEDIUM + REPAIRABLE sont recommandées et doivent être
corrigées lorsque possible sans créer de nouveau conflit.

Le résultat est un remplacement complet du Weekly Plan AI Output V1,
jamais un patch ou un diff.

10. Revalidate schema, semantics, normalization, pool, business rules and analytics
11. Run the required final review or deterministic final checks
12. Save only if every required layer passes
13. If still invalid, fail gracefully without persistence

Limite V1:

* max 1 repair attempt
* si impossible, retourner une erreur contrôlée
* ne jamais sauvegarder un plan invalide

Plus tard:

* max 2 repair attempts
* fallback model plus fort
* possibilité de demander précision utilisateur si contexte insuffisant
